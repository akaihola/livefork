## Configuration

livefork reads `.livefork.toml` from the repository root. Every key has a default; the file can be as minimal as you like.

```toml
# .livefork.toml

[upstream]
remote = "upstream"          # git remote pointing to the original project  (default: "upstream")
branch = "main"              # upstream branch to track                      (default: "main")

[fork]
remote = "origin"            # your fork's remote                            (default: "origin")
branch = "main"              # your fork's main branch                       (default: "main")

[knit]
branch = "johndoe"           # name of the git-knit merge branch
                             # default: local username ($USER / getpass.getuser())
                             # using the username avoids clashing with upstream branch names
base   = "main"              # base branch for git-knit  (default: same as fork.branch)

[fork-readme]
enabled = true               # maintain a fork README in the main branch     (default: true)
file    = "README.md"        # file to keep updated                          (default: README.md)
push    = true               # push origin/main after each update            (default: true)
# template = ".livefork/readme-template.md"   # optional custom Jinja2 template

[[branches]]
name = "feature/my-patch-1"
description = "Fix awkward error message in CLI"
# push defaults to true when the branch tracks origin/<name>,
# false for purely local branches

[[branches]]
name = "feature/my-patch-2"
description = "Add --dry-run flag to delete command"
pr = "https://github.com/upstream-org/project/pull/42"  # set by `livefork pr create`

[[branches]]
name = "private/integration"
description = "Integration with internal tooling (not for upstream)"
push = false                 # override: never push this branch
```

### Branch auto-detection

When no `[[branches]]` sections are present, `livefork init` and `livefork create` scan the local repository and include every branch that belongs to you:

| Branch tracking configuration         | Included? | `push` default |
| ------------------------------------- | --------- | -------------- |
| Tracks `origin/<name>` (your fork)    | ✓ yes     | `true`         |
| No upstream configured (purely local) | ✓ yes     | `false`        |
| Tracks `upstream/<name>`              | ✗ no      | –              |
| Tracks any other remote               | ✗ no      | –              |

The branch name is used as the description. Edit `.livefork.toml` afterwards to refine descriptions or override `push`.

## The sync workflow

`livefork sync` executes these steps in order:

```
Step 1 │ Fetch upstream
       │   git fetch <upstream.remote>
       │
Step 2 │ Advance your fork's main
       │   git checkout <fork.branch>
       │   git reset --hard <upstream.remote>/<upstream.branch>
       │   (discards the previous fork README commit, if any)
       │
Step 3 │ Rebase each topic branch   (one by one, in config order)
       │   git checkout <branch>
       │   git rebase <fork.branch>
       │   → if conflict: pause, print instructions, exit 1
       │
Step 4 │ Rebuild the merge branch
       │   git knit rebuild --working-branch <knit.branch>
       │
Step 5 │ Regenerate fork README  (if fork-readme.enabled)
       │   Render README.md listing upstream info, sync date, and all branches
       │   git checkout <fork.branch>
       │   git add <fork-readme.file>
       │   git commit -m "chore: update fork README [skip ci]"
       │   git push --force-with-lease <fork.remote> <fork.branch>
       │
Step 6 │ Done ✓
```

Progress is saved after each successful step so that `livefork continue` resumes exactly where the process paused.

The fork's main branch is always **exactly one commit ahead** of upstream/main: the fork README commit. Because the README is regenerated by resetting to upstream/main and re-layering it, `--force-with-lease` is used for the push and `git reset --hard` replaces `git merge --ff-only`.

### Rebase strategy

Each branch is rebased with `git rebase --rebase-merges` to preserve any merge structure. `git rerere` is enabled repo-locally so resolutions are remembered and replayed automatically on future syncs.

## Conflict resolution

When a rebase conflict occurs, `livefork sync` pauses and prints:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠  Rebase conflict on feature/my-patch-2
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Rebasing: feature/my-patch-2
     on:  main  (after merging upstream/main @ a3f91c2)

Conflicting commit:
  d7b4e1a  Add --dry-run flag to delete command

Conflicting files:
  src/cli.py  (content conflict)

To resolve manually:
  1. Open the conflicting file(s) listed above.
  2. Edit, keeping the changes you want.
  3. git add <file> ...
  4. livefork continue

To abort this sync entirely:
  livefork abort

To resolve with a coding agent:
  livefork agent-context > /tmp/conflict.md
  # Hand /tmp/conflict.md to your AI assistant, then run:
  livefork continue
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Manual resolution

```bash
# 1. Open conflicting files and resolve the conflict markers
$EDITOR src/cli.py

# 2. Stage the resolved files
git add src/cli.py

# 3. Resume – livefork completes the rebase and all remaining steps
livefork continue
```

### Coding-agent-assisted resolution

`livefork agent-context` dumps a self-contained Markdown document with everything an AI assistant needs:

```bash
livefork agent-context > /tmp/conflict.md
```

The document contains:

- Which branch is being rebased and onto what
- The full diff of the conflicting commit (what your branch was trying to do)
- The upstream diff that caused the conflict (what changed in main since the branch was cut)
- The current state of each conflicting file with conflict markers
- Exact shell commands to stage and continue

Hand it to your coding assistant:

```bash
claude "$(cat /tmp/conflict.md)"
# or open it in Cursor, Codex, etc.
```

Once the agent has resolved the conflict and staged the files:

```bash
livefork continue
```

livefork verifies that all conflict markers are gone and the index is clean before resuming.

## Pull request draft workflow

Each topic branch may contain a `PULL-REQUEST-DRAFT.md` file at the repository root. The first line is the PR title; everything after the first blank line is the PR body.

```markdown
Add --dry-run flag to the delete command

The `delete` subcommand currently removes items immediately without
any preview. This can be destructive if run by mistake.

This patch adds a `--dry-run` / `-n` flag that prints what would be
deleted without actually deleting anything.

## Testing

Run `myproject delete --dry-run` and verify that nothing is deleted.
```

While the draft exists, `livefork status` shows `[draft]` next to the branch and the fork README links directly to `PULL-REQUEST-DRAFT.md` on GitHub. After `livefork pr create`, the fork README links to the submitted PR instead.

### Lifecycle

```
branch created
    │
    ▼
livefork draft feature/my-patch-2   ← open editor, write PULL-REQUEST-DRAFT.md
    │
    ▼
(iterate: commit fixes, update draft as needed)
    │
    ▼
livefork pr create feature/my-patch-2
    │  reads title + body from PULL-REQUEST-DRAFT.md
    │  runs: gh pr create --title "..." --body "..."
    │  records PR URL in .livefork.toml
    │  removes PULL-REQUEST-DRAFT.md, commits the removal
    │  regenerates fork README (so the PR link appears immediately)
    ▼
branch merged / closed upstream
    │
    ▼
livefork remove feature/my-patch-2
```

## Fork README

When `fork-readme.enabled = true` (the default), livefork maintains a `README.md` in the fork's main branch. It is regenerated on every `livefork sync` and pushed, so the previous version is always discarded – there is no meaningful history to preserve.

The fork README serves three goals:

1. **Clarity** – make it immediately clear to any visitor that this is a personal fork, not the canonical upstream project.
2. **Navigation** – link prominently to the upstream repository.
3. **Branch inventory** – list every topic branch with a link to its current status: a submitted PR, the `PULL-REQUEST-DRAFT.md`, or the branch on GitHub.

### Example output

```markdown
# project (personal fork – @johndoe)

> ⚠️ This is a personal fork of
> [upstream-org/project](https://github.com/upstream-org/project).
> The `main` branch tracks upstream and contains no other changes.
> Personal modifications live in the branches listed below.

## Upstream

|             |                                                                            |
| ----------- | -------------------------------------------------------------------------- |
| Repository  | [upstream-org/project](https://github.com/upstream-org/project)            |
| Synced to   | `main` @ [a3f91c2](https://github.com/upstream-org/project/commit/a3f91c2) |
| Last synced | 2026-03-18                                                                 |

## Personal branches

| Branch                    | Description                          | Status               |
| ------------------------- | ------------------------------------ | -------------------- |
| [feature/my-patch-1][b1]  | Fix awkward error message in CLI     | [draft PR][b1-draft] |
| [feature/my-patch-2][b2]  | Add --dry-run flag to delete command | [PR #42][pr42]       |
| [private/integration][b3] | Integration with internal tooling    | local only           |

[b1]: https://github.com/johndoe/project/tree/feature/my-patch-1
[b1-draft]: https://github.com/johndoe/project/blob/feature/my-patch-1/PULL-REQUEST-DRAFT.md
[b2]: https://github.com/johndoe/project/tree/feature/my-patch-2
[pr42]: https://github.com/upstream-org/project/pull/42
[b3]: https://github.com/johndoe/project/tree/private/integration
```

Branches with `push = false` are listed as "local only" and no GitHub link is generated.

### Custom template

Supply a Jinja2 template to override the default layout:

```toml
[fork-readme]
template = ".livefork/readme-template.md"
```

The template receives a context object with `upstream`, `fork`, `branches`, `knit`, and `synced_at` fields. See the built-in template in `src/livefork/templates/fork-readme.md.j2` for reference.

## Command reference

### `livefork create <repo>`

Fork a GitHub repository, clone it, write `.livefork.toml`, and initialise the merge branch – all in one step. Requires [GitHub CLI][gh-cli].

```bash
livefork create upstream-org/project
livefork create https://github.com/upstream-org/project

# Flags (core ones mirror gh repo fork)
  --fork-name        NAME    # name for the fork on GitHub         (default: same as upstream)
  --owner            OWNER   # GitHub user or org to own the fork  (default: authenticated user)
  --clone-path       PATH    # local directory                     (default: ./<repo-name>)
  --upstream-remote  NAME    # name for the upstream remote        (default: "upstream")
  --fork-remote      NAME    # name for the fork remote            (default: "origin")
  --merge-branch     NAME    # override merge branch name          (default: $USER)
  --no-init                  # skip .livefork.toml and git-knit
```

When `--owner` names an organisation, `gh repo fork --org` is used. When it names your own GitHub account (or is omitted), the fork goes to your personal account – the default `gh` behaviour. Passing another user's name is rejected with a clear error, since GitHub does not support forking into someone else's account.

### `livefork init`

Configure an existing fork clone and initialise the merge branch.

```bash
livefork init
  --upstream         URL     # upstream remote URL (prompted if ambiguous)
  --merge-branch     NAME    # override merge branch name          (default: $USER)
```

### `livefork sync`

Fetch upstream → rebase all branches → rebuild merge branch → push fork README.

```bash
livefork sync
  --dry-run                  # show what would happen without doing it
  --branch           NAME    # rebase one branch only
  --no-knit                  # skip the git-knit rebuild step
  --no-readme                # skip the fork README step
```

### `livefork continue`

Resume a paused sync after resolving a conflict. Completes the current rebase, then proceeds with remaining branches, the knit rebuild, and the fork README push.

### `livefork abort`

Abort a paused sync. Resets every already-rebased branch to its pre-sync tip using the saved progress state.

### `livefork status`

Show branch sync state and PR status.

```
upstream/main  a3f91c2  Add streaming support to API client
fork/main      a3f91c2  ✓  in sync with upstream  (fork README up to date)

TOPIC BRANCHES
  feature/my-patch-1      d2a3f01  ✓  rebased  (3 ahead)   [draft]
  feature/my-patch-2      7c19e3a  ✓  rebased  (1 ahead)   [PR #42 submitted]
  private/integration     e841fa9  ✗  needs rebase          [no draft]

MERGE BRANCH
  johndoe                 c908b4d  ✓  up to date (all topic branches present)
```

### `livefork draft <branch>`

Create or edit `PULL-REQUEST-DRAFT.md` on a branch. Commits the file automatically after the editor exits (only if changed).

```bash
livefork draft feature/my-patch-1
  --message  TEXT    # set the title line; open editor for the body only
```

### `livefork pr create <branch>`

Submit a pull request from `PULL-REQUEST-DRAFT.md`. Records the PR URL in `.livefork.toml`, removes the draft file, and regenerates the fork README. Requires [GitHub CLI][gh-cli].

```bash
livefork pr create feature/my-patch-1
  --base   BRANCH    # target branch on upstream  (default: upstream.branch)
  --draft            # open as a draft PR on GitHub
```

### `livefork readme`

Regenerate and push the fork README on demand (step 5 of `livefork sync` in isolation).

```bash
livefork readme
  --no-push          # regenerate without pushing
  --dry-run          # print generated content without writing or pushing
```

### `livefork agent-context`

Print a structured Markdown conflict report for a coding AI assistant. See [Coding-agent-assisted resolution](#coding-agent-assisted-resolution).

### `livefork add <branch>`

Add a topic branch to the configuration and to the merge branch.

```bash
livefork add feature/new-thing
  --description  TEXT
```

### `livefork remove <branch>`

Remove a topic branch from the configuration and from the merge branch.

### `livefork knit`

Rebuild the merge branch without touching topic branches. Equivalent to `git knit rebuild --working-branch <knit.branch>`.

## State file

livefork stores transient sync progress in `.git/livefork-state.json`. The file is created at the start of `sync`, updated after each step, and deleted when the sync completes or is aborted. It is human-readable JSON if you need to inspect it.

## Tips and best practices

**Keep topic branches small and focused.** Small branches rebase cleanly and produce focused, reviewable diffs. Split large branches with `git rebase -i` before syncing.

**Write the PR draft early.** Create `PULL-REQUEST-DRAFT.md` as soon as you start a branch – even a rough title is enough. The draft is visible in the fork README and in `livefork status` from day one.

**Use descriptive `description` fields.** The description appears in `livefork status` and in the agent context document. A clear one-liner helps you – and an AI assistant – understand each branch when resolving a conflict.

**Let `git rerere` do the heavy lifting.** livefork enables `rerere` automatically. After resolving a conflict once, Git records the resolution and replays it on future syncs. To share resolutions across machines, commit `.git/rr-cache` or set `rerere.autoupdate = true`.

**Sync frequently.** The less upstream has moved since your last sync, the simpler any conflicts will be. Weekly is much easier than quarterly.

**Pin branch order in config.** `git-knit` merges branches in the order they appear in `.livefork.toml`. Put branches that others depend on earlier in the list.

## Relationship to `git-knit`

| Tool       | Responsibility                                                         |
| ---------- | ---------------------------------------------------------------------- |
| `git-knit` | Build and maintain a merge branch from a set of topic branches         |
| `livefork` | Track an upstream, keep topic branches rebased, orchestrate `git-knit` |

`livefork` calls `git-knit` internally for the merge-branch step. You can also use `git knit` commands directly at any time – `livefork` reads the standard `git-knit` configuration from `.git/config`.
